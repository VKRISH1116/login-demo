Login Demo — The Complete Guide

A from-scratch authentication system: React + TypeScript → Spring Boot → PostgreSQL, using a stateless JWT delivered in an httpOnly cookie. This document explains what we used, what else we could have used, why we chose what we did, and walks through every line of code — then gives you a script to present it.

Spring Boot 4.1Java 21React 19 + VitePostgreSQL 16JWT (jjwt 0.12.6)BCrypt

How to read this doc Parts 1–2 are the "story" you tell. Part 3 is your reference when someone points at a line and asks "what does that do?". Part 4 is the actual talk track. If you only have 20 minutes to prepare, read Part 1, the two tables in Part 2, and the demo script.

1 · What we built

A user can register, log in, see a protected page that greets them by email, stay logged in across a page refresh, and log out. That short list quietly exercises every hard part of authentication.

The 4 things the system does

  1. Register — store an account with a hashed password.
  2. Log in — verify the password, hand back a signed token in a cookie.
  3. Stay logged in — the browser resends the cookie; the server trusts the signature, not a memory of you.
  4. Log out — server tells the browser to delete the cookie.

The 5 threats it defends against

  • Password theft from the DB → BCrypt one-way hashing.
  • Token theft via XSS → httpOnly cookie (JS can't read it).
  • Cross-site request forgery → SameSite=Lax cookie.
  • User enumeration → identical 401 for "no such user" and "wrong password".
  • Cross-origin abuse → CORS locked to one exact origin.

2 · Core security concepts (say these in your own words)

Before any code, these are the ideas the whole project is built on. If you can explain these five, you understand the project.

Authentication vs Authorization

Authentication (AuthN) = "who are you?" (proving identity — the login). Authorization (AuthZ) = "what are you allowed to do?" (permissions). This demo is almost entirely authentication; the only authorization is the blunt rule "you must be logged in to see /api/me." No roles yet.

HTTP is stateless

Every HTTP request is independent — the server remembers nothing between requests by default. So after you log in, the very next request has no idea who you are. Something must travel with each request to re-prove identity. That "something" is our token. Two classic ways to carry it:

  • Server-side sessions: server stores a session in memory/DB, gives the browser a random session-id cookie. Stateful — the server must remember every logged-in user.
  • Tokens (JWT): the proof itself travels with the request and is self-verifying via a signature. Stateless — the server stores nothing. We chose this.

What a JWT actually is

A JWT is just a string of three Base64URL parts joined by dots: header.payload.signature.

  • Header — the algorithm, e.g. {"alg":"HS512"}.
  • Payload — the "claims": who (sub=email), when issued (iat), when it expires (exp). This is only Base64, not encrypted — anyone can read it. Never put secrets here.
  • Signature — HMAC-SHA512(header.payload, secret). Only someone with the server's secret can produce it. Change one character of the payload and the signature no longer matches → the token is rejected.
The key insight to say out loudThe server doesn't "remember" tokens. When a token comes back, the server re-computes the signature with its secret. If it matches, the token is authentic and un-tampered. That's how it stays stateless.

Why hash passwords with BCrypt (not store them, not "encrypt" them)

We never store the password. We store a BCrypt hash — a one-way function. You can go password→hash, never hash→password. On login we hash the attempt and compare. BCrypt also (a) mixes in a random salt so two identical passwords produce different hashes, and (b) is deliberately slow, so brute-forcing a stolen database is expensive. Encryption would be the wrong tool — encryption is reversible; we want irreversibility.

The XSS ↔ CSRF ↔ cookie trade-off (the crux of this project)

Where you store the token decides which attack you're exposed to:

Store token in…Exposed to XSS?Exposed to CSRF?Notes
localStorageYes — JS reads itNoAny injected script can steal the token. Easy to code.
httpOnly cookie (our choice)No — JS can't read itYes → mitigateBrowser attaches it automatically; we add SameSite=Lax to blunt CSRF.

We deliberately took the harder path (httpOnly cookie) because XSS token theft is silent and total, whereas CSRF has a well-known, cheap mitigation (SameSite). This choice is why the project also needs CORS-with-credentials — see the code.

5 · Alternatives we could have used — and why we didn't

This table is gold in a presentation: it shows you made decisions, not defaults.

Decision pointWhat we pickedAlternativesWhy we chose ours
Token storage httpOnly cookie localStorage / sessionStorage; in-memory JS variable localStorage is readable by any XSS script → silent token theft. A cookie with HttpOnly is invisible to JS. We accepted the CSRF surface because SameSite handles it cheaply.
Session strategy Stateless JWT Server-side sessions (Spring Session + Redis) No session store to run or scale; the token verifies itself. Trade-off: you can't instantly revoke a JWT (mitigated by short 15-min expiry).
JWT signing HMAC (HS512), symmetric RSA/ECDSA (RS256), asymmetric One service issues and verifies the token, so one shared secret is simplest. Asymmetric keys matter when a different service must verify without being able to sign.
Database PostgreSQL in Docker H2 in-memory; MySQL; SQLite We wanted a real, persistent, production-like DB (H2 vanishes on restart). Docker keeps it isolated and reproducible. Postgres on port 5433 to avoid clashing with a native Postgres 17.
Password hash BCrypt Argon2, PBKDF2; SHA-256; plaintext BCrypt is the well-supported default in Spring Security. Argon2 is arguably stronger but adds config; plain SHA-256 is too fast (brute-forceable); plaintext is negligence.
Build tool (FE) Vite Create-React-App; Webpack; Next.js Vite is fast and minimal. Next.js would add SSR/routing we don't need for a demo. CRA is effectively deprecated.
DTOs Java records Classes with Lombok; plain POJOs Records are immutable, boilerplate-free, and built into Java 21 — perfect for request/response shapes.
Honest trade-offs to mention (shows maturity)
  • JWTs can't be instantly revoked. If a token leaks, it's valid until it expires. We limit the blast radius with a 15-minute lifetime and no refresh token yet.
  • We disabled CSRF protection in Spring for now because the SPA flow needs a matching token strategy; the cookie's SameSite=Lax is our current CSRF defense. Turning Spring's CSRF back on is a listed next step.
  • Secret is in application.properties — fine for dev, but in production it must come from an environment variable / secret manager.

We use package-by-feature: user/ (the data), auth/ (login logic), config/ (security wiring). Here's every file in the order data flows.

7.1 · The data — user/User.java

👤 User.java JPA entity
@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(name = "password_hash", nullable = false)
    private String passwordHash;

    @Column(name = "created_at", nullable = false)
    private Instant createdAt = Instant.now();

    protected User() {}   // JPA requires a no-arg constructor

    public User(String email, String passwordHash) {
        this.email = email;
        this.passwordHash = passwordHash;
    }
    // getters only — no setters, so a User is effectively immutable after creation
}
  1. @Entity + @Table("users") — tells Hibernate "this class is a row in the users table." Because of ddl-auto=update, the table is created from these fields.
  2. @Id + @GeneratedValue(IDENTITY) — id is the primary key; the database auto-increments it.
  3. email @Column(unique=true) — a DB-level uniqueness constraint: two users can't share an email even if the app check is bypassed.
  4. passwordHash — note the name: we store the hash, never the password. Column is NOT NULL.
  5. createdAt = Instant.now() — audit timestamp, set at object creation.
  6. protected no-arg constructor — JPA/Hibernate constructs entities reflectively and needs this; protected discourages app code from using it.
  7. Getters only, no setters — once created, a User doesn't change here, which prevents accidental mutation.

7.2 · The queries — user/UserRepository.java

🗄️ UserRepository.java Spring Data JPA
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email);
    boolean existsByEmail(String email);
}
  1. extends JpaRepository<User, Long> — inheriting this gives us save(), findById(), delete(), etc. for free. User = entity type, Long = id type.
  2. findByEmail — we write only the method name; Spring parses "findBy + Email" and generates SELECT * FROM users WHERE email = ?. Returns Optional (may be empty) so we're forced to handle "not found."
  3. existsByEmail — generates a cheap SELECT COUNT / exists query; used to reject duplicate registrations.
Talking pointThis interface has zero implementation and yet works. That's the magic of Spring Data: derived query methods. It's the single most impressive "look how little code" moment in the demo.

7.3 · The data shapes — DTOs (records)

📨 RegisterRequest.java / LoginRequest.java / LoginResponse.java / MeResponse.java
public record RegisterRequest(
    @Email @NotBlank String email,
    @NotBlank @Size(min = 8, message = "Password must be at least 8 characters")
    String password
) {}

public record LoginRequest(@Email @NotBlank String email, @NotBlank String password) {}
public record LoginResponse(String email) {}
public record MeResponse(String email) {}
  1. record — a compact, immutable data carrier. One line replaces a whole class with fields, constructor, getters, equals/hashCode.
  2. @Email @NotBlank — validation runs before your controller code. A malformed email → automatic 400 Bad Request, you write no if-statements.
  3. @Size(min=8) on register only — server-side enforcement of the password rule. The frontend checks too, but the server is the real gate (never trust the client).
  4. Separate Request vs Response records — the client never sends a password hash, and we never return one. Different shapes for different directions = safer API.

7.4 · The token engine — auth/JwtService.java

🔑 JwtService.java create & verify JWTs
@Service
public class JwtService {

    private final SecretKey key;
    private final long expirationMs;

    public JwtService(@Value("${app.jwt.secret}") String secret,
                      @Value("${app.jwt.expiration-ms}") long expirationMs) {
        this.key = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
        this.expirationMs = expirationMs;
    }

    public String generateToken(String email) {
        Date now = new Date();
        Date expiry = new Date(now.getTime() + expirationMs);
        return Jwts.builder()
                .subject(email)      // "sub" claim = who
                .issuedAt(now)       // "iat"
                .expiration(expiry)  // "exp"
                .signWith(key)       // HMAC signature (HS512, inferred from key size)
                .compact();          // → "header.payload.signature"
    }

    public String extractEmail(String token) {
        Claims claims = Jwts.parser()
                .verifyWith(key)
                .build()
                .parseSignedClaims(token)  // throws if signature wrong OR expired
                .getPayload();
        return claims.getSubject();
    }
}
  1. @Service — registers this as a Spring-managed singleton so it can be injected anywhere.
  2. @Value("${app.jwt.secret}") — Spring pulls the secret and expiry straight from application.properties into the constructor.
  3. Keys.hmacShaKeyFor(secret.getBytes(...)) — turns the text secret into a cryptographic key. Because the secret is >512 bits, jjwt upgrades to HS512 automatically.
  4. generateToken: compute now and expiry = now + 15min, then build claims (sub=email, iat, exp), sign with the key, and compact() into the dotted string.
  5. extractEmail: verifyWith(key) + parseSignedClaims re-checks the signature and expiry. If the token was tampered with or is expired, this throws — which is exactly how the filter knows to reject it.
  6. getSubject() — pulls the email back out of the verified payload.

7.5 · The security wiring — config/SecurityConfig.java

🛡️ SecurityConfig.java the rulebook
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

@Bean
public SecurityFilterChain filterChain(HttpSecurity http,
        JwtAuthenticationFilter jwtAuthenticationFilter) throws Exception {
    http
        .cors(Customizer.withDefaults())                       // ① enable CORS (bean below)
        .csrf(csrf -> csrf.disable())                          // ② off for now (SPA + SameSite)
        .sessionManagement(sm -> sm.sessionCreationPolicy(STATELESS))  // ③ no server sessions
        .authorizeHttpRequests(auth -> auth
            .dispatcherTypeMatchers(DispatcherType.ERROR).permitAll()  // ④ let errors render
            .requestMatchers("/api/auth/**").permitAll()          // ⑤ login/register public
            .anyRequest().authenticated()                      // ⑥ everything else needs auth
        )
        .exceptionHandling(e -> e.authenticationEntryPoint(
            new HttpStatusEntryPoint(HttpStatus.UNAUTHORIZED)))    // ⑦ return 401, not a login redirect
        .addFilterBefore(jwtAuthenticationFilter,
            UsernamePasswordAuthenticationFilter.class);       // ⑧ run OUR filter early
    return http.build();
}

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("http://localhost:5173"));   // exact origin, not "*"
    config.setAllowedMethods(List.of("GET","POST","PUT","DELETE","OPTIONS"));
    config.setAllowedHeaders(List.of("Content-Type"));
    config.setAllowCredentials(true);                            // allow the cookie
    // ...register this config for every path /**
}
  1. passwordEncoder() bean → BCrypt — one bean, injected wherever we hash/verify passwords.
  2. ① .cors(...) — turns on CORS using the config bean at the bottom. Needed because :5173 and :8080 are different origins.
  3. ② .csrf(disable) — deliberately off for this SPA phase; SameSite=Lax on the cookie is our current CSRF defense. (Re-enabling is a next step.)
  4. ③ STATELESS — tells Spring "never create an HTTP session." This enforces our JWT design; identity must come from the token every time.
  5. ④ ERROR dispatch permitAll — lets Spring's error pages render so a 400 doesn't get masked into a confusing 401/403.
  6. ⑤ /api/auth/** permitAll — register & login must be reachable without a token (chicken-and-egg: you can't have a token before you log in).
  7. ⑥ anyRequest().authenticated() — the default-deny rule: everything else (like /api/me) requires a valid authentication.
  8. ⑦ HttpStatusEntryPoint(401) — for an API we want a clean 401, not Spring's default redirect to a login HTML page.
  9. ⑧ addFilterBefore(jwtFilter, ...) — inserts our JWT filter early in the chain so identity is established before the authorization checks run.
  10. CORS bean — setAllowedOrigins("http://localhost:5173") is an exact origin (a wildcard * is illegal once setAllowCredentials(true) is set — the browser refuses to send cookies to a wildcard origin). This pairing is the whole reason CORS was a milestone.

7.6 · The gatekeeper — auth/JwtAuthenticationFilter.java

🚧 JwtAuthenticationFilter.java runs on every request
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain filterChain) {

        String token = readTokenCookie(request);                // pull "token" cookie

        if (token != null && SecurityContextHolder.getContext()
                                    .getAuthentication() == null) {
            try {
                String email = jwtService.extractEmail(token);  // verify sig + expiry
                var auth = new UsernamePasswordAuthenticationToken(
                        email, null, Collections.emptyList());
                auth.setDetails(new WebAuthenticationDetailsSource()
                        .buildDetails(request));
                SecurityContextHolder.getContext().setAuthentication(auth);  // "logged in"
            } catch (Exception e) {
                // bad/expired token → stay anonymous → protected routes 401
            }
        }
        filterChain.doFilter(request, response);                // continue the chain
    }
}
  1. extends OncePerRequestFilter — guarantees this runs exactly once per HTTP request.
  2. readTokenCookie(request) — loops the request's cookies looking for one named token; returns its value or null.
  3. if token != null && authentication == null — only try to authenticate if there's a token and nobody's authenticated yet.
  4. jwtService.extractEmail(token) — the verification step. If the token is forged or expired it throws, and we fall into the empty catch → the request stays anonymous.
  5. new UsernamePasswordAuthenticationToken(email, null, emptyList()) — builds Spring's "this request is authenticated as email" object. null credentials (we don't keep the password), empty list = no roles.
  6. SecurityContextHolder…setAuthentication(auth) — this is the moment the user becomes "logged in" for this request. Downstream, anyRequest().authenticated() now passes.
  7. empty catch — intentional: an invalid token isn't an error to shout about; it just means "not authenticated," and the 401 happens naturally later.
  8. filterChain.doFilter(...) — hand off to the next filter/controller either way.

7.7 · The endpoints — auth/AuthController.java

🎯 AuthController.java /api/auth/*
@PostMapping("/register")
public ResponseEntity<Void> register(@Valid @RequestBody RegisterRequest req) {
    if (userRepository.existsByEmail(req.email()))
        return ResponseEntity.status(CONFLICT).build();          // 409 duplicate
    String hash = passwordEncoder.encode(req.password());        // BCrypt hash
    userRepository.save(new User(req.email(), hash));
    return ResponseEntity.status(CREATED).build();               // 201
}

@PostMapping("/login")
public ResponseEntity<LoginResponse> login(@Valid @RequestBody LoginRequest req) {
    User user = userRepository.findByEmail(req.email()).orElse(null);
    if (user == null || !passwordEncoder.matches(req.password(), user.getPasswordHash()))
        return ResponseEntity.status(UNAUTHORIZED).build();      // 401 — same for both cases

    String token = jwtService.generateToken(user.getEmail());
    ResponseCookie cookie = ResponseCookie.from("token", token)
            .httpOnly(true)          // JS cannot read it (XSS defense)
            .secure(false)          // dev only; MUST be true over HTTPS in prod
            .path("/")              // sent on every path
            .sameSite("Lax")        // CSRF mitigation
            .maxAge(Duration.ofMillis(jwtExpirationMs))
            .build();
    return ResponseEntity.ok()
            .header(HttpHeaders.SET_COOKIE, cookie.toString())
            .body(new LoginResponse(user.getEmail()));
}

@PostMapping("/logout")
public ResponseEntity<Void> logout() {
    ResponseCookie cookie = ResponseCookie.from("token", "")  // empty value
            .httpOnly(true).secure(false).path("/").sameSite("Lax")
            .maxAge(0)               // expire immediately → browser deletes it
            .build();
    return ResponseEntity.noContent()
            .header(HttpHeaders.SET_COOKIE, cookie.toString()).build();  // 204
}
  1. @RestController @RequestMapping("/api/auth") — every method here lives under /api/auth; return values become JSON.
  2. Constructor injection — UserRepository, PasswordEncoder, JwtService, and the expiry value are injected by Spring (no new, no globals).
  3. register → @Valid — validation fires first; bad input never reaches your logic.
  4. existsByEmail → 409 — duplicate email is rejected with 409 Conflict.
  5. passwordEncoder.encode — BCrypt-hashes the password (with a random salt) before saving. The plaintext never touches the DB.
  6. login → findByEmail(...).orElse(null) — look the user up.
  7. the combined if — "no such user OR wrong password" both return the same 401. This prevents user enumeration (an attacker can't discover which emails exist).
  8. passwordEncoder.matches(raw, hash) — BCrypt re-hashes the attempt with the stored salt and compares. We never decrypt anything.
  9. ResponseCookie … httpOnly(true) — the security-defining line: the browser stores it but JavaScript literally cannot read document.cookie for it.
  10. secure(false) — allowed only because dev is plain http://localhost; in production this must be true so the cookie is HTTPS-only.
  11. sameSite("Lax") — the browser won't attach this cookie on most cross-site requests → CSRF mitigation.
  12. maxAge = 15 min — cookie lifetime matches the JWT expiry.
  13. logout — you can't delete an httpOnly cookie from JS, so the server overwrites it with an empty value and maxAge=0, telling the browser to drop it. Returns 204 No Content.

8 · Frontend — line by line

8.1 · The API layer — src/api.ts

🌐 api.ts fetch wrapper
const BASE = "http://localhost:8080";

async function request<T>(path, options = {}): Promise<T> {
  const res = await fetch(`${BASE}${path}`, {
    method: options.method ?? "GET",
    credentials: "include",                     // ★ send & receive the cookie
    headers: { "Content-Type": "application/json" },
    body: options.body,
  });
  if (!res.ok) throw new ApiError(res.status);  // non-2xx → typed error
  const text = await res.text();
  return (text ? JSON.parse(text) : undefined);
}

export function register(email, password) { return request("/api/auth/register", {method:"POST", body: JSON.stringify({email,password})}); }
export function login(...)  { return request("/api/auth/login", ...); }
export function me()       { return request("/api/me"); }
export function logout()   { return request("/api/auth/logout", {method:"POST"}); }
  1. credentials: "include" — the single most important line in the frontend. Without it, fetch will neither send nor store cross-origin cookies, and login would silently "work" but every following request would be anonymous.
  2. ApiError(res.status) — we throw a typed error carrying the HTTP status so pages can say "401 → wrong password" vs "409 → email taken."
  3. text ? JSON.parse : undefined — handles endpoints like logout (204) that return an empty body without crashing JSON.parse.
  4. Four thin exports — the rest of the app calls login(), me(), etc. and never touches fetch details again.

8.2 · The auth brain — src/auth/AuthContext.tsx

🧠 AuthContext.tsx global auth state
export function AuthProvider({ children }) {
  const [email, setEmail] = useState<string|null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {                          // runs once on app load
    apiMe()                                  // "who am I?" using whatever cookie exists
      .then(res => setEmail(res.email))      // 200 → logged in
      .catch(() => setEmail(null))         // 401 → not logged in
      .finally(() => setLoading(false));
  }, []);

  async function login(e, p) { const res = await apiLogin(e,p); setEmail(res.email); }
  async function logout()   { await apiLogout(); setEmail(null); }

  return <AuthContext.Provider value={{email, loading, login, logout}}>{children}</>;
}
  1. useState(email, loading) — the app's single source of truth for "who is logged in" and "are we still checking."
  2. useEffect(..., []) — the refresh-persistence mechanism. On every fresh page load it calls /api/me. The browser still holds the httpOnly cookie, so the server answers 200 with your email — and you're logged in again without re-entering your password. This is exactly the behaviour you verified.
  3. catch → setEmail(null) — a 401 (no/expired cookie) just means "not logged in," handled quietly.
  4. loading flag — critical: until /api/me answers we don't yet know the auth state, so the route guard must wait (see 8.3) rather than flash the login page.
  5. login() / logout() — call the API then update state; because state lives in context, the whole UI re-renders in sync.

8.3 · The route guard — src/routes/ProtectedRoute.tsx

🚦 ProtectedRoute.tsx
export function ProtectedRoute({ children }) {
  const { email, loading } = useAuth();
  if (loading) return <p>Loading…</p>;         // wait for /api/me before deciding
  if (!email)  return <Navigate to="/login" replace />;  // not logged in → bounce
  return <>{children}</>;                     // logged in → show the page
}
  1. if (loading) — while the initial /api/me is in flight we render a placeholder. Skipping this would flash "/login" for a split second on every refresh even when logged in.
  2. if (!email) → Navigate to /login — no user → redirect. replace avoids polluting browser history.
  3. else render children — authenticated, so show the protected page.

11 · Q&A preparation

Why a cookie instead of localStorage?
localStorage is readable by any JavaScript, so one XSS bug = silent token theft. An httpOnly cookie is invisible to JS. The downside is CSRF, which we mitigate with SameSite=Lax.
If it's stateless, how do you log someone out or ban them?
You can't instantly invalidate a JWT — that's the honest cost of statelessness. We limit exposure with a 15-minute expiry. Real revocation needs a token blocklist or short-lived access + refresh tokens (a planned next step).
What stops me from editing the JWT payload to become another user?
The signature. Change one byte of the payload and HMAC-SHA512(payload, secret) no longer matches; parseSignedClaims throws and the filter rejects it. You can't forge a valid signature without the server's secret.
Isn't the JWT payload readable? Isn't that insecure?
It's Base64, not encryption — anyone can read the email and expiry. That's fine: a JWT proves authenticity, not secrecy. We never put anything sensitive in it. Password lives only as a BCrypt hash in the DB.
Why the same 401 for "no such user" and "wrong password"?
To prevent user enumeration. Different messages would let an attacker discover which emails are registered by probing.
Why did CORS need special configuration?
The frontend (:5173) and backend (:8080) are different origins, so the browser enforces CORS. And because we send cookies, we must set allowCredentials(true) — which forbids a * wildcard origin, so we list the exact origin http://localhost:5173.
Why BCrypt and not SHA-256?
SHA-256 is fast — great for checksums, bad for passwords because attackers can try billions/sec. BCrypt is deliberately slow and salted, so cracking a stolen DB is expensive and rainbow tables don't work.
What would you change for production?
secure(true) on the cookie (HTTPS-only), the JWT secret from an env var/secret manager, versioned DB migrations (Flyway) instead of ddl-auto=update, re-enable Spring CSRF, add refresh tokens and rate-limiting on login.
Why HS512 (symmetric) and not RS256 (asymmetric)?
One service both issues and verifies the token, so a single shared secret is simplest and fastest. Asymmetric keys matter when other services must verify tokens without the power to mint them.
